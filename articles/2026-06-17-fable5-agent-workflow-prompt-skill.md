# 当 AI Agent 开始自己发明工作流：Prompt 和 Skill 真的要过时了吗？

过去一年，很多人学习 AI 的方式，都是围绕几个关键词展开的：`Prompt`、`Workflow`、`Skill`、`SOP`。

我们试图把任务拆细，把步骤写清，把上下文补全，把模型可能犯错的地方提前堵住。某种程度上，人类是在给 AI 搭脚手架：告诉它先做什么、后做什么、遇到问题怎么判断、什么情况下该停下来问人。

但 Claude Fable 5 出现后，一些早期体验者开始提出一个有点残酷的问题：

如果模型自己已经会规划、会拆解、会调用工具、会验证、会反思，甚至会临时发明新的工具链，那我们之前学的那些流程技巧，还剩多少价值？

## 从“教它怎么做”到“告诉它什么算完成”

Simon Willison 记录了一个很典型的例子。

他在自己的项目里发现一个很小的前端问题：一个输入框下面出现了不该有的横向滚动条。他把截图丢给 Claude Code 里的 Fable 5，只给了一句提示：去依赖里看看原因。

然后他离开电脑做家务。

回来时，他发现浏览器自己被打开了。Fable 5 不是简单地读代码、猜原因，而是自己启动本地开发服务，在 Playwright、Firefox、Safari 等环境里尝试复现问题。复现不了时，它又写临时 HTML 页面、调用 macOS 的窗口截图能力、改模板注入 JavaScript、搭了一个本地 CORS 服务，把浏览器里的测量数据回传到文件里。

最后的修复是什么？

两行 CSS。

这件事震撼的地方不在于两行 CSS，而在于中间的路径。没有人告诉它“请你写一个截图工具”“请你做一个本地数据回传服务”“请你修改模板模拟快捷键”。它只是为了完成目标，自己判断缺什么，然后补什么。

这正是 Simon 用来形容 Fable 5 的词：`relentlessly proactive`，不依不饶地主动。

## 人类越来越像“甲方”

Ethan Mollick 的体验则更进一步。

他让 Fable 5 做一张基于真实交通数据的等时线地图：用户选择城市后，可以看到从这个城市出发，在不同时间内能到达世界哪些地方。这个任务看似是可视化，实际需要研究航班、火车、机场通勤、步行、驾车、偏远地区交通等大量细节。

Fable 5 的做法是：启动多个子 Agent 做研究，检索数千条交通数据，同时自己写代码、做验证、记录进展。Mollick 后来指出偏远地区不能只估算，要找真实路线。模型又启动了互相挑错的 Agent 流程，去补皮特凯恩岛、加拿大北极地区等偏远地点的数据。

另一个案例里，Mollick 让它解决一个研究中的实际问题：如何校准人类评分员与 AI 判断之间的差异。Fable 5 先写了 19 页设计文档，然后连续运行九个半小时，交付出一套名为 Concord 的软件工具。

Mollick 的感受很微妙：一方面很兴奋，因为他只是提出要求，东西就做出来了；另一方面也不安，因为他只是提出要求，东西就做出来了。

这说明人和 AI 的关系正在变化。过去我们像“巫师”：写咒语，期待模型响应。现在更像“甲方”：描述目标，支付成本，验收结果。真正的过程发生在黑箱里，模型自己做了大量细小判断，而人未必看得见，也未必能逐步介入。

## 旧 Prompt 不是没用，而是重心变了

所以，Prompt、Skill、Workflow 真的要过时了吗？

更准确的说法是：旧式的“过程型提示词”正在贬值。

以前我们写提示词，常常是在教模型怎么做事：

- 先调研，再列计划，再实现，再测试；
- 先输出 A，再等待确认，再输出 B；
- 遇到异常时按某个分支处理。

这些方法在模型能力较弱时很有用，因为模型缺少稳定的任务管理能力，人类必须帮它搭框架。

但当模型本身已经能规划、能调用工具、能派出子 Agent、能做验证时，过细的流程反而可能变成限制。你以为是在帮助它，其实可能是在把它锁进人类过去总结出来的旧路径里。

新的关键输入，可能不再是“怎么做”，而是：

- 什么是目标？
- 什么是边界？
- 什么算完成？
- 哪些风险不能接受？
- 结果如何验证？

换句话说，Prompt 没死，Skill 也没死。它们从“操作手册”变成了“验收标准”和“约束协议”。

## 真正重要的是验收能力

这对使用 AI 的人提出了更高要求。

以前会写提示词的人，优势在于能把流程拆清楚。以后更重要的能力，可能是定义问题、识别风险、设计验收。

你不一定要告诉模型每一步怎么走，但你必须知道它交出来的东西是否可靠。

比如代码任务里，不能只说“帮我修好”，而要说明：

- 修复后应该通过哪些测试；
- 哪些行为不能改变；
- 哪些文件或模块不要碰；
- 如果涉及权限、网络、数据安全，需要在哪些地方停下来确认；
- 最终结果要如何复现和验证。

这类输入不是低级提示词，而是高质量工程判断。

当 Agent 越来越强，人类的价值不再是把每一步都写成 SOP，而是能判断什么问题值得做、什么方案可接受、什么结果不能上线。

## 更强的 Agent，也意味着更高的风险

Simon 的案例还有另一层警示。

Fable 5 为了调试一个 CSS 问题，可以打开真实浏览器、修改模板、调用系统截图、搭本地服务、读取回传数据。这很强，但也很危险。

如果这样的 Agent 在没有沙箱的环境里运行，如果它读到恶意 issue、恶意网页、恶意文档里的提示注入，它同样可能用这种“主动性”去做不该做的事。

模型越聪明，越会绕路；模型越主动，越需要边界。

所以未来的 Agent 工作流，不只是“少写步骤、多写目标”，还必须包含权限控制、沙箱隔离、可审计日志、敏感操作确认和自动化测试。

否则，主动性就是双刃剑。

## 结论：不是白学了，而是该升级了

Fable 5 带来的启发不是“Prompt 和 Skill 都白学了”。

真正的变化是：人类不再需要把所有步骤都写给模型，但更需要把目标、边界和验收写清楚。

旧时代的 Prompt，是教 AI 怎么做。

新时代的 Prompt，是定义什么叫做好。

旧时代的 Skill，是把人类流程封装给模型。

新时代的 Skill，应该更像可验证的工作契约：输入是什么，约束是什么，成功标准是什么，失败时如何停下来。

AI 时代最残酷的事，也许不是学不会。

而是你刚学会一种操作技巧，模型就把它自动化了。

但这并不意味着人没有价值。恰恰相反，当执行越来越便宜，判断就越来越贵。下一阶段真正稀缺的，不是会不会写一段漂亮提示词，而是能不能提出正确问题，并知道什么时候可以相信答案。
